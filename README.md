# 書籍管理システム

書籍管理システムのバックエンドAPI。

このドキュメントはプロジェクトの制約・要件を定義するものであり、実装（GenAI/エージェントによる実装を含む）の際の参照前提とする。
要件は **必須要件（テスト要件）** と **独自追加要件** に分けて記載する。

---

## 1. 必須要件（テスト要件）

> これらはコーディングテストとして必ず満たすべき要件。
> 要件が満たされていれば独自ライブラリの追加は可。ただし**不要な機能の実装などのオーバーエンジニアリングは減点対象**。

### 1.1 スコープ

**やること**
- APIの実装
- テスト（可能な範囲で）

**やらないこと**
- フロントエンドの実装

**規約**
- GenAI利用可（エージェント機能も可）

### 1.2 概要

- 書籍管理システム

### 1.3 技術スタック（必須）

- Kotlin
- Spring Boot
- jOOQ

### 1.4 機能

| 機能 | エンドポイント |
| --- | --- |
| 書籍の登録 | `POST /api/books` |
| 書籍の更新 | `PUT /api/books/{bookId}` |
| 著者の登録 | `POST /api/authors` |
| 著者の更新 | `PUT /api/authors/{authorId}` |
| 著者に紐づく書籍情報の取得 | `GET /api/authors/{authorId}/books` |

### 1.5 ドメインモデル

**書籍（books）**
- タイトル
- 価格（**0以上**であること）
- 著者（**最低1人**の著者を持つ。複数の著者を持つことが可能）
- 出版状況（`未出版` / `出版済み`。**出版済みステータスのものを未出版には変更できない**）

**著者（authors）**
- 名前
- 生年月日（**現在日以前**であること）
- 著者も複数の書籍を執筆できる

> 書籍と著者は **多対多** の関係。

### 1.6 評価観点

- 指定された技術スタックの適用
- フレームワークやライブラリの適切な利用
- 実行可能性
- 仕様に沿った動作
- 変数名やクラス名、関数名が実態を明確に反映しているか
- Null安全性
- コードフォーマットの整合性
- 変数に再代入しないなど、ベストプラクティスの遵守
- オーバーエンジニアリングしていないか
- 適切な単体テストが作成されているか

### 2.2 OpenAPI / スキーマ駆動

> **API定義は OpenAPI スキーマを唯一の正とする（スキーマ駆動・型安全を必須とする）。**

- **サーバ側コード生成**: [OpenAPI Generator](https://openapi-generator.tech/) の `kotlin-spring` ジェネレータを採用。Gradle 連携は `org.openapi.generator` プラグインを使用する。
  - `interfaceOnly=true` とし、**API interface と DTO のみ生成**する（Controller 実装やSpringボイラープレートは生成しない）。実装側は生成 interface を実装する。
- **API クライアント / ドキュメント表示**: [Scalar](https://github.com/scalar/scalar) を採用。OpenAPI スキーマの描画に加え、**手動疎通テスト用の API クライアントとしても利用**する（Postman 等の代替を兼ねるため別途クライアントは導入しない）。

### 2.3 jOOQ コード生成パイプライン

- jOOQ のコード生成は **ローカルの PostgreSQL（Docker Compose）を前提**とする。Testcontainers ベースの生成は Gradle 環境では手間が大きいため採用しない。
- 生成フローは **Flyway でスキーマ適用 → jOOQ がそのスキーマからコード生成** の順とする。
- 生成コードはコミット対象とし、通常のビルド・テストは DB 起動を不要とする。スキーマ変更時のみ DB を起動して `./gradlew jooqCodegen` で再生成し、差分をコミットする。

### 2.4 設計原則

- レイヤードアーキテクチャ + ドメインレイヤー
- **出版状況の遷移などのビジネスルールはドメイン層で検証する**（Controller / Service に散らさない）。

### 2.5 HTTPステータス方針

| ステータス | 用途 |
| --- | --- |
| `200 OK` | GET など取得成功 |
| `201 Created` | POST などリソース作成成功 |
| `400 Bad Request` | バリデーションエラー（入力形式不正） |
| `404 Not Found` | リソースなし |
| `422 Unprocessable Entity` | ビジネスエラー（不正な更新など） |

> バリデーションエラー（入力値の形式・制約違反）は `400`、
> ドメインルール違反（例: 出版済み→未出版への変更）は `422` として区別する。
> **エラーレスポンスボディは作り込まず、Spring Boot 標準のエラーハンドリングに委ねる。**

### 2.6 更新・参照に関する確定事項

- **更新は PUT（リソース全体の置換）**とする。部分更新（PATCH）は導入しない。
- **書籍と著者は ID 参照**で関連付ける。リクエストで指定された ID（パスの `{bookId}`/`{authorId}`、および書籍登録時の著者ID）が存在しない場合は **`404`** を返す。
- 著者は 0 冊でも登録可能（著者単独で登録できる）。書籍は登録・更新時に **最低 1 人の著者ID** を必須とする。

### 2.7 DBスキーマ

> ドメインモデル（1.5）と多対多の関係を反映した PostgreSQL スキーマ。
> DDL は Flyway マイグレーション（`src/main/resources/db/migration/`）で管理し、これを jOOQ のコード生成元とする（2.3 参照）。
> 価格・生年月日・出版状況の遷移、および文字列長などの**入力制約は DB に定義せず、ドメイン層で検証する**（2.4 / [ADR](docs/adr/README.md) 参照）。
> このため文字列カラムは長さ制限を持たない `text` 型とする。

#### テーブル一覧

| テーブル | 役割 |
| --- | --- |
| `authors` | 著者 |
| `books` | 書籍 |
| `author_books` | 著者と書籍の多対多を表す中間テーブル |

#### `authors`

| カラム | 型 | 説明 |
| --- | --- | --- |
| `id` | `bigint` | 著者ID（PK・自動採番） |
| `name` | `text` | 名前 |
| `birth_date` | `date` | 生年月日 |
| `created_at` | `timestamptz` | 作成日時 |
| `updated_at` | `timestamptz` | 更新日時 |

#### `books`

| カラム | 型 | 説明 |
| --- | --- | --- |
| `id` | `bigint` | 書籍ID（PK・自動採番） |
| `title` | `text` | タイトル |
| `price` | `integer` | 価格 |
| `publication_status` | `text` | 出版状況（`UNPUBLISHED` / `PUBLISHED`） |
| `created_at` | `timestamptz` | 作成日時 |
| `updated_at` | `timestamptz` | 更新日時 |

#### `author_books`（中間テーブル）

| カラム | 型 | 説明 |
| --- | --- | --- |
| `author_id` | `bigint` | 著者ID（FK → `authors(id)`） |
| `book_id` | `bigint` | 書籍ID（FK → `books(id)`） |

- 主キーは `(author_id, book_id)` の複合主キー（同一ペアの重複を防止）。先頭列が `author_id` のため、唯一のリレーション参照系である `GET /api/authors/{authorId}/books`（著者→書籍）が主キーインデックスだけで完結する。
- 「書籍は最低1人の著者を持つ」関係は中間テーブルだけでは表現できないため、ドメイン層で担保する。
- 書籍→著者方向の補助インデックスは設けない（参照系の要件がなく、過剰実装となるため。[ADR](docs/adr/README.md) 参照）。
