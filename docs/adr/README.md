# アーキテクチャ決定記録（ADR）

技術的な決定事項を1行ずつ記録する。新しい決定は末尾に追記する。

| # | 日付 | 決定 | 理由 |
| --- | --- | --- | --- |
| 1 | 2026-06-27 | 文字列カラムは `varchar(n)` ではなく `text` を採用し、文字数などの入力制約はドメイン層で検証する | PostgreSQL では両者に性能差がなく、`varchar(255)` 等の長さは根拠の薄い慣習値。入力制約はドメイン層で検証する方針（README 2.4）と一貫させる |
| 2 | 2026-06-27 | 中間テーブルは `author_books`、複合主キーを `(author_id, book_id)` とする | 唯一のリレーション参照系 `GET /api/authors/{authorId}/books`（著者→書籍）が主キーインデックスだけで完結する |
| 3 | 2026-06-27 | 中間テーブルに書籍→著者方向（`book_id`）の補助インデックスは作らない | 存在チェックは各テーブルのPK、重複防止は複合PKで足り、`book_id` を使うのは低頻度の書き込みのみ。参照系の要件もなく過剰実装となるため |
| 4 | 2026-06-27 | `INSERT ... RETURNING` での採番ID取得は `fetchOne()` ではなく `fetchSingle()` を使う | INSERT 直後は必ず1件返り、失敗時は例外が飛ぶ。`fetchOne()`（0件で null）だと到達不能な null ガードが必要になるため、1件を前提とする `fetchSingle()` で意図を明示する |
| 5 | 2026-06-27 | 中間テーブルへの複数リンク挿入は、行を1つずつ足すループではなく `valuesOfRows` で全行をまとめて渡す単一の複数行 INSERT を使う | jOOQ が案内する全行まとめ渡しの書き方に沿い、ループや変数の受け取り直し（再代入）が不要になる。著者数は数件規模でラウンドトリップも1回 |
| 6 | 2026-06-27 | リポジトリ層のテストは `@JooqTest` スライス＋Testcontainers（Singleton Container）で行い、フルコンテキスト（`@SpringBootTest`）から分離する | `@JooqTest` は永続化層のみをロードしテストごとに自動ロールバックする。jOOQ は方言依存の SQL を生成するため組み込み DB へ差し替えず実 PostgreSQL を使う（H2 では検証にならない）。コンテナは JVM 内で1つ共有し起動コストを抑える |
